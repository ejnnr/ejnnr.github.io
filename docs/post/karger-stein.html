Trading off speed against the probability of success in the Karger-Stein Algorithm

December 6, 2020
Graphs

(Note: this is an analysis of one aspect of the Karger-Stein algorithm, it’s not meant to be a beginner-friendly introduction)

Karger’s algorithm randomly contracts a graph and surprisingly, this can be used to find a minimum cut with probability O(n−2)\mathcal{O}(n^{-2})O(n−2), where nnn is the number of vertices of the graph. This is a much, much higher probability than sampling a graph cut uniformly at random would give! But it still means we need to run the algorithm O(n2log⁡n)\mathcal{O}(n^2\log n)O(n2logn) times to get a high success probability. Karger’s algorithm can be implemented in O(n2)\mathcal{O}(n^2)O(n2) time1, which gives an overall runtime of O(n4log⁡n)\mathcal{O}(n^4 \log n)O(n4logn) — not great, there are much faster deterministic algorithms.

To understand how the Karger-Stein algorithm improves upon that, we need the following key result that forms the foundation for Karger’s algorithm:

Theorem2: When Karger’s algorithm contracts a graph from nnn to rrr vertices, any given mincut survives with probability ≥(r2)/(n2)=r(r−1)n(n−1)\geq {r \choose 2}/{n \choose 2} = \frac{r (r - 1)}{n (n - 1)}≥(2r​)/(2n​)=n(n−1)r(r−1)​.

In particular, for r=2r = 2r=2, we get the O(n−2)\mathcal{O}(n^{-2})O(n−2) probability mentioned above.

But note the following: if we make only a few contractions, r≲nr \lesssim nr≲n, then mincuts are almost guaranteed to survive! This is the key insight that allows us to improve the runtime of Karger’s algorithm, leading to the improved Karger-Stein algorithm.

The idea is the following: first we contract the graph down to roughly nb\frac{n}{b}bn​ vertices, where bbb is small enough that mincuts are very likely to survive. Then we branch: we again contract the graph down by a factor of bbb, but we do so aaa times independently from one another. aaa needs to be chosen high enough that mincuts are very likely to survive in at least one of the branches. We repeat this process until we have contracted the graph down to just 2 vertices3. If we chose aaa and bbb right, at least one of the final leaves of our computational tree will likely contain a mincut. So we return the best cut we’ve found among all the leaves.

The Karger-Stein algorithm as it was originally described and as it is usually presented uses a=2a = 2a=2 and b=2b = \sqrt{2}b=2​. So we always split the computation into two branches and reduce the number of vertices by a factor of 2\sqrt{2}2​ before branching again. But in this post, I would like to motivate where these numbers come from, as well as show that they’re not the only ones that work. So in the following, we’re going to analyze the “generalized Karger-Stein algorithm” with arbitrary aaa and bbb.

Success probability

As mentioned above, any minimum cut survives a contraction from nnn to rrr vertices with probability ≥(r2)/(n2)=r(r−1)n(n−1)\geq {r \choose 2}/{n \choose 2} = \frac{r (r - 1)}{n (n - 1)}≥(2r​)/(2n​)=n(n−1)r(r−1)​. I said we first contract to “roughly” nb\frac{n}{b}bn​ vertices — to be precise we contract until ⌈nb+1⌉\lceil \frac{n}{b} + 1\rceil⌈bn​+1⌉ vertices are left, this will give us a nice bound. The probability that a mincut survives this contraction is4:

p≥⌈nb+1⌉⌈nb⌉n(n−1)≥(nb+1)⋅nbn(n−1)≥1b2\begin{split} p &\geq \frac{\lceil \frac{n}{b} + 1 \rceil \lceil \frac{n}{b} \rceil}{n (n - 1)}\\ &\geq \frac{(\frac{n}{b} + 1) \cdot \frac{n}{b}}{n (n - 1)}\\ &\geq \frac{1}{b^2} \end{split}p​≥n(n−1)⌈bn​+1⌉⌈bn​⌉​≥n(n−1)(bn​+1)⋅bn​​≥b21​​

We can now apply this bound recursively: After we have contracted to ⌈nb+1⌉\lceil \frac{n}{b} + 1 \rceil⌈bn​+1⌉ vertices, we can forget that this is a partially contracted graph, and just treat this number as the “new nnn“.

We will write pkp_kpk​ for the survival probability if there are kkk levels of recursion left before we reach the leaves of the tree. So p0p_0p0​ will be the probability in the leaves of the recursion tree. Depending on when precisely we stop the recursion and what method we use to finish the contraction, p0p_0p0​ might take different values, but all that matters for us is that it is some constant.

Using the bound we found above, we get the following recurrence:

pk+1≥1−(1−pkb2)ap_{k + 1} \geq 1 - \left( 1 - \frac{p_k}{b^2} \right)^apk+1​≥1−(1−b2pk​​)a

What’s going on here? pkb2\frac{p_k}{b^2}b2pk​​ is a lower bound on the probability that any given mincut survives in one particular branch. So (1−pkb2)a\left(1 - \frac{p_k}{b^2}\right)^a(1−b2pk​​)a is an upper bound on the probability that the mincut survives in none of the aaa branches, and consequently 1−(1−pkb2)a1 - \left( 1 - \frac{p_k}{b^2} \right)^a1−(1−b2pk​​)a is a lower bound on the probability that it survives in at least one.

where we used zk≥1z_k \geq 1zk​≥1 in the last step. The constant term may depend on aaa and bbb but not on zkz_kzk​.

If a≥b2a \geq b^2a≥b2, then zk∈O(k)z_k \in \mathcal{O}(k)zk​∈O(k) which means that pk∈Ω(1k)p_k \in \Omega\left(\frac{1}{k}\right)pk​∈Ω(k1​). The depth of recursion for a graph with nnn vertices is Θ(log⁡n)\Theta(\log n)Θ(logn), so the overall success probability is Ω(1log⁡n)\Omega\left(\frac{1}{\log n}\right)Ω(logn1​).

What this means in words: if we create enough branches (at least b2b^2b2) compared to how long we contract before branching again, then we get quite a high success probability — Ω(1log⁡n)\Omega\left(\frac{1}{\log n}\right)Ω(logn1​) means that log⁡2n\log^2 nlog2n runs are enough to get an overall success probability that approaches 1 as n→∞n \to \inftyn→∞.

But what if a<b2a < b^2a<b2, i.e. if we don’t have enough branches at each stage? Then the inequality derived above only yields zk∈O((b2a)k)z_k \in \mathcal{O}\left(\left(\frac{b^2}{a}\right)^k\right)zk​∈O((ab2​)k) so the success probability pkp_kpk​ can be exponentially low in kkk. This means we’d have to repeat the algorithm a potentially exponential number of times, which would make it useless.

That still leaves the question: why does the Karger-Stein algorithm use a=b2a = b^2a=b2 in particular, when a>b2a > b^2a>b2 would give a success probability at least as high? For that we need to turn to the runtime complexity.

Runtime

The runtime of the Karger-Stein algorithm can be described with the following recurrence:

T(n)=aT(nb)+O(n2)T(n) = aT\left(\frac{n}{b}\right) + \mathcal{O}(n^2)T(n)=aT(bn​)+O(n2)

The O(n2)\mathcal{O}(n^2)O(n2) term is for contracting down to roughly nb\frac{n}{b}bn​ vertices. At that point, we solve aaa smaller version of the original problem, each of size nb\frac{n}{b}bn​. That leads to the aT(nb)aT\left(\frac{n}{b}\right)aT(bn​) term.

This kind of recurrence is exactly what the Master theorem is for. In this case, if we choose a=b2a = b^2a=b2, we get a runtime of Θ(n2log⁡n)\Theta(n^2 \log n)Θ(n2logn) for a single run of the Karger-Stein algorithm. We already saw that with a<b2a < b^2a<b2, we get an exponentially low success probability, so that choice isn’t interesting anyway. Finally, if a>b2a > b^2a>b2, we get a high success probability, but the runtime becomes Θ(nc)\Theta(n^c)Θ(nc), where c:=log⁡alog⁡b>2c := \frac{\log a}{\log b} > 2c:=logbloga​>2.

We can summarize all our results (and a few I didn’t mention) in one table:

The “total runtime” column contains the runtime that is needed to achieve a high success probability by repeating the Karger-Stein algorithm often enough (at least if nnn is large enough). This is the complexity that we want to minimize in practice.

We can now see that combining the analysis of the success probability with the runtime analysis explains why the Karger-Stein algorithm uses a=b2a = b^2a=b2: in the other cases, we are either very unlikely to succeed and therefore need too many runs, or we are taking unnecessarily long for a single run.

But also note that any choice of a “branching factor” aaa works, as long as we then choose b=ab = \sqrt{a}b=a​. So splitting the computation up into just two subproblems is a reasonable and simple choice, but from a purely asymptotic perspective it is arbitrary.
